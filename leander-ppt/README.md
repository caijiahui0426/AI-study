# Leander PPT Skill 使用说明

`leander-ppt` 是一个用于生成、重构、标准化和打磨 PowerPoint 材料的 Codex Skill。它不是单纯的“PPT 美化提示词”，而是一套带有大纲、主题、脚手架、组件、章节隔离和渲染自检的 PPT 生产流程。

适用场景包括：

- 内部领导汇报
- 产品介绍
- 项目复盘或阶段汇报
- 内部分享材料
- 对外客户演示
- 现有 PPT 标准化改版
- 从 PRD、文档、纪要、材料包生成正式 PPT

## 安装与调用

将整个 `leander-ppt` 文件夹放到 Codex 的 skills 目录下：

```text
C:\Users\<你的用户名>\.codex\skills\leander-ppt
```

目录中必须至少包含：

```text
leander-ppt/
+-- SKILL.md
+-- agents/
+-- references/
+-- templates/
```

安装后重启 Codex Desktop，再新开线程。在输入框中输入 `/`，应能看到 `leander-ppt`。

如果看不到：

- 先确认文件夹名称就是 `leander-ppt`。
- 确认 `SKILL.md` 顶部 frontmatter 里的 `name` 是 `leander-ppt`。
- 完全退出并重启 Codex Desktop，而不是只新开线程。
- 确认没有同时安装插件版、命令版或重复 skill 目录。

## 推荐使用方式

可以这样触发：

```text
使用 leander-ppt，基于这份材料生成一份内部领导汇报 PPT，先出大纲，不要直接生成 PPT。
```

或者：

```text
/leander-ppt 帮我把这份产品介绍材料整理成正式 PPT，先按 skill 流程走，主题和样稿需要我确认。
```

如果已有参考 PPT，应明确说明：

```text
参考这份已有 PPT 的页面范式和视觉风格，按 leander-ppt 流程重新生成一版。
```

## 核心流程

这个 skill 默认按以下流程工作：

1. **Brief + Outline**  
   先理解材料、 audience、场景、目标，生成 `brief.md` 和 `outline.md`。

2. **大纲确认**  
   确认章节结构、每页标题、展示形式、素材边界和样稿页。

3. **主题确认**  
   明确使用哪个主题或模板。不能在没有主题确认的情况下直接生成样稿。

4. **实例化脚手架**  
   必须建立可复用的 PPT 项目结构，包括 `theme/`、`components/`、`chapters/`、`output/preview/` 等。

5. **Anchor Sample**  
   先生成 2-3 页真实可编辑样稿，并渲染成图片检查。

6. **用户确认样稿**  
   样稿确认后，才能进入完整 PPT 生产。

7. **分章节或批次生产**  
   多章节材料要按物理章节隔离，每个章节有自己的 `chapter.json`。

8. **渲染自检和修复**  
   输出 PPTX 后必须渲染成 PNG/contact sheet，检查页面是否符合模板、是否越界、是否错乱、是否过空。

## 必须遵守的规则

以下规则不要绕过：

- 不要一上来就生成完整 PPT。
- 不要跳过主题确认。
- 不要用散装临时脚本生成多页 PPT。
- 多章节材料必须先划分物理章节。
- 每页要能追溯到 `outline.md` 和 `chapter.json`。
- 每页要说明组件来源、素材来源和数据边界。
- Anchor sample 必须来自正式脚手架，而不是一次性探索稿。
- 输出完成前必须渲染预览图并自检。
- 如果预览图不像已确认模板，即使没有重叠或越界，也算 QA 失败。
- 如果工具链降级或使用替代方案，必须写入 `qa.md`，不能假装是最终质量。

## 模板和组件

`leander-ppt` 支持三类模板/组件来源：

- **内置脚手架组件**：位于 `templates/leander-ppt-scaffold/components/`。
- **内部 PPT 抽象组件**：从已有优秀 PPT 中沉淀标题、卡片、矩阵、流程、机制图、看板、图文页等范式。
- **外部渲染组件**：例如 ECharts、Three.js、Mapbox、Rive、Spline 等，最终以高清 PNG/SVG 插入 PPT，并保留源文件。

原则上，能用可编辑 PPT 形状表达的内容，优先用可编辑形状；只有真实截图、地图、复杂图表、3D 场景等才建议作为图片或外部渲染插入。

## 字体和基础风格

默认字体规范：

- 中文：微软雅黑 / Microsoft YaHei
- 英文和数字：Century Gothic

默认视觉方向不是固定一种颜色，而是根据主题 token 决定。当前内置 `Leander Base` 更偏正式浅色汇报风格，可复用红色强调、深蓝结构色、浅色背景和清晰标题系统。

## 质量检查重点

检查 PPT 时，不只看“有没有生成出来”，还要看：

- 页面是否像已确认模板。
- 内容是否过空或过密。
- 字体是否可读。
- 图形、图片、文字是否都在页面安全区内。
- 页面视觉重心是否偏向一侧。
- 每页是否有真实视觉解释，而不是只有文字卡片。
- 中文是否乱码。
- 图片、图标、图表是否真的传递信息。
- 结论、数据、状态是否有来源边界。
- 修改是否只影响目标页，没有误伤其他页面。

## 适合分享时强调的点

这个 skill 的重点不是“让 AI 自动做漂亮 PPT”，而是让 AI 按稳定流程和可复用结构来做 PPT：

- 先定逻辑，再定主题。
- 先做样稿，再铺全量。
- 先隔离章节，再并行或批量。
- 先渲染检查，再说完成。
- 发现问题后按最小单页或组件修复。

如果使用者只想快速出一版草稿，也可以让 Codex 使用较快模式，但需要明确接受质量风险。

## 常见问题

**为什么它会停下来问我确认？**  
这是设计行为。主题、样稿和大纲不确认，后面整套 PPT 很容易返工。

**为什么不直接生成完整 PPT？**  
因为 PPT 的风险通常不在“能不能生成”，而在主题跑偏、结构跑偏、页面范式跑偏。先做样稿可以提前暴露问题。

**为什么要有脚手架？**  
脚手架让主题、组件、章节、页面和 QA 都有固定位置。没有脚手架，后续很难做最小单页修复，也难以复用模板能力。

**为什么需要渲染预览？**  
PPTX 文件结构正常不代表页面看起来正常。必须渲染成图片才能发现错位、遮挡、留白、字号和模板偏差。

**如果 `/` 里出现两个 leander-ppt 怎么办？**  
通常是同时存在 skill、plugin 或 command 版本。保留 `C:\Users\<用户名>\.codex\skills\leander-ppt` 这一份即可，删除重复入口后重启 Codex Desktop。

## 分享前检查清单

分享给别人前，建议确认：

- [ ] 只保留一个 `leander-ppt` skill 目录。
- [ ] `SKILL.md` frontmatter 正常。
- [ ] `agents/openai.yaml` 存在。
- [ ] `references/` 文件完整。
- [ ] `templates/leander-ppt-scaffold/` 存在。
- [ ] 没有包含本地项目的敏感材料、客户材料或临时输出。
- [ ] 示例模板不包含不能外传的 Logo、数据、截图或客户信息。

## 维护建议

后续如果继续打磨这个 skill，优先沉淀三类东西：

- 真实可复用组件：标题页、章节页、四象限、流程图、机制图、数据看板、图文页、路线图。
- 主题 token：颜色、字体、间距、安全区、Logo 位置、标题系统。
- QA 规则：哪些情况必须判定失败，哪些情况可以作为低置信度草稿交付。

每次真实任务跑偏后，不要只改当次 PPT，最好同步判断是否需要更新 `references/` 或 scaffold 组件。
